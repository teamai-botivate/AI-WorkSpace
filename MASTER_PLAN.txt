================================================================================
  SECTION 1: IDEA / VISION
================================================================================

1. Ye pura AI Workspace ek WHITE-LABEL PRODUCT hai.
   - Multiple companies ko poora code ke saath dena hai.
   - Company apne server pe deploy karegi ya hum unke liye karenge.

2. Setup process DEAD SIMPLE hona chahiye.
   - Ek baar company apne credentials setup kare → saare agents kaam karne lagein.
   - Ek command, ek form — bas.

3. Completely DYNAMIC — kisi bhi company ke liye kaam kare.
   - Koi bhi company apne Google Account, OpenAI Key, SMTP Email lagaye → chalu.

4. Hamara KUCH BHI hardcoded nahi hoga.
   - Na hamari API keys, na hamara Google account, na hamara email.
   - Companies apna sab kuch daalegi.

5. HR Support ka flow SIMPLE banana hai.
   - Registration process hataana hai.
   - Pahli baar open kare → Direct setup: Google Sheet connect, Policies upload,
     Employee passwords generate.
   - Uske baad → Sirf Login page, employees apne ID/password se login karke
     chat karein.

6. MODULAR — kisi ko ek agent dena ho ya do, wo bhi chale.
   - Sirf HR Support dena ho → de do.
   - Sirf Resume Screener dena ho → de do.
   - Dono dena ho → de do.
   - Setup tab bhi sirf ussi agent ke credentials maange.

────────────────────────────────────────────────────────────────────────────────
  PROBLEM 1: Google OAuth Client ID HARDCODED in Frontend Code
────────────────────────────────────────────────────────────────────────────────

  WHERE:
    HR_Support/frontend/src/pages/OnboardingPage.jsx (Line 88)
    → const clientId = "561628688869-uhibvk67hslomtphrq8hp4ha1ktviefl..."

  WHY IT'S BAD:
    Ye HAMARA Google Cloud Project ka Client ID hai.
    Har company ka apna Google Cloud Project hoga.
    Code me hamara ID committed hai → company change nahi kar sakti easily.

  SOLUTION:
    - Frontend environment variable se load karo: VITE_GOOGLE_CLIENT_ID
    - Ye value root .env se aayegi → setup wizard me company apna daalegi
    - Code me koi bhi Google ID string nahi hoga

  PROBLEM 4: Koi Unified Setup Process Nahi Hai
────────────────────────────────────────────────────────────────────────────────

  CURRENT STATE:
    Company ko setup karne ke liye:
    1. HR_Support/backend/.env manually edit karo (8-10 values)
    2. Resume-Screening-Agent/.env manually edit karo (3-4 values)
    3. Google Cloud Console me manually OAuth setup karo

    5. Har agent ki venv banaao, dependencies install karo
    6. HR Support me jaake company register karo (OnboardingPage)
    7. Tab jaake kuch kaam karega

  WHY IT'S BAD:
    - 7 steps, 3 files, manually → bahut tedious
    - Non-technical company ke liye impossible
    - Galti hone ke chances bahut zyada

  SOLUTION:
    - Ek setup.ps1 (or setup.py) wizard banaao
    - Interactive prompts: Company Name? OpenAI Key? SMTP Email? Google IDs?
    - Wizard automatically:
      → Root .env generate kare
      → workspace.config.json update kare
      → venvs create kare + dependencies install kare
      → node_modules install kare
      → Validity check kare (test API keys)
    - Ek command: .\setup.ps1 → sab ho jaaye

  PROBLEM 6: Agents Tightly Coupled — Ek Agent Alag Se Nahi De Sakte
────────────────────────────────────────────────────────────────────────────────

  CURRENT STATE:
    - workspace.config.json me saare agents hardcoded hain
    - Agar sirf HR Support dena ho to Resume Screening ka code bhi jaayega
    - start-dev.ps1 sab kuch start karta hai (no selection)
    - Setup wizard (when built) sab agents ke credentials maangega

  SOLUTION:
    Agent Manifest System:
    - Har agent folder me ek agent.manifest.json hoga
    - Gateway auto-discovers agents by scanning folders for this file
    - workspace.config.json me sirf enabled agents ki list hogi
    - setup wizard manifest padh ke sirf required credentials maange
    - start-dev.ps1 sirf enabled agents start kare

    Example agent.manifest.json:
    {
      "id": "hr-support",
      "name": "HR Employee Support",
      "requires": {
        "credentials": ["OPENAI_API_KEY", "GOOGLE_OAUTH_CLIENT_ID", "SMTP_USER"],
        "optional": ["GROQ_API_KEY"]
      },
      "backend": {
        "entrypoint": "app.main:app",
        "port": 8001,
        "healthCheck": "/health",
        "runtime": "python"
      },
      "frontend": {
        "type": "vite",
        "devPort": 5175
      }
    }

────────────────────────────────────────────────────────────────────────────────
  PROBLEM 10: 5 Ports Open Hain — Confusing
────────────────────────────────────────────────────────────────────────────────

  CURRENT:
    :3000  → Main Frontend Shell
    :9000  → Gateway API
    :8000  → Resume Screening
    :8001  → HR Support Backend
    :5175  → HR Support Frontend
    
    Company ko 5 ports samajhne padenge. Firewall rules, CORS issues, confusion.

  SOLUTION — SINGLE PORT ARCHITECTURE:
    Gateway (:9000 dev, :3000 prod) becomes the ONLY port users interact with.
    
    Dev Mode:
      Vite (:3000) proxies /api/* → Gateway (:9000)
      Gateway proxies /agent/hr-support/api/* → :8001
      Gateway proxies /agent/hr-recruiter/api/* → :8000
      Agent frontends still run on own ports (iframe)
      → User only opens localhost:3000
    
    Production Mode:
      Gateway (:3000) serves EVERYTHING:
        /              → Main frontend (built static)
        /api/*         → Gateway own APIs
        /agent/{id}/api/* → Reverse proxy to agent backends
        /agent/{id}/*  → Agent frontend (built static)
      → User only opens localhost:3000 (or company domain)
